# Onboarding — datacomns-api-automation

Welcome. This repo is an **API test framework**: shared libraries under `framework/`, one folder per API under `projects/`.

## Day 1 — run something

1. **Python 3.10+** installed.
2. From repo root:

   ```bash
   cd datacomns-api-automation
   python3 -m venv .venv
   source .venv/bin/activate   # Windows: .venv\Scripts\activate
   pip install -r requirements.txt
   cp .env.example .env
   ```

3. **DCC smoke** (needs `https://dcc-qa.ccdi.cancer.gov` reachable unless you skip live tests):

   ```bash
   bash scripts/run_smoke.sh
   ```

   Or a single file:

   ```bash
   pytest projects/dcc/tests/api_smoke/test_DCC_TC01_verify_organizations.py -v
   ```

   If you are offline: `export DATACOMNS_SKIP_LIVE_TESTS=1` and live tests skip.

4. **Federation OpenAPI contract** (needs `DATACOMNS_FEDERATION_BASE_URL` in `.env`):

   ```bash
   bash scripts/run_federation_contract_suite.sh
   ```

## Mental model

| Path | Purpose |
|------|---------|
| `framework/` | **Do not duplicate** — HTTP client, config loader, assertions, AI context helpers |
| `config/projects.yaml` | **Single registry** — base URLs per project (env overrides) |
| `projects/<slug>/` | One API product; add `conftest.py` when you have pytest tests |
| `projects/dcc/tests/api_smoke/` | DCC fast checks (`@pytest.mark.smoke`) |
| `projects/dcc/tests/memgraph_api/` | DCC Memgraph + REST pairs |
| `projects/federation/contracts/` | Federation OpenAPI spec for the contract runner (no pytest here by default) |

## Writing your first test

1. Open `projects/dcc/tests/api_smoke/test_DCC_TC01_verify_organizations.py` as a pattern.
2. Use the `api_client` fixture from `projects/dcc/conftest.py` and `api_client.get(...)`.
3. Use `assert_successful_json(response)` from `framework.assertions`.
4. Add `@pytest.mark.smoke` or `@pytest.mark.regression`.

Paths are **relative to** `api_prefix` (e.g. DCC uses `/api/v1` + your path).

## Commands cheat sheet

| Goal | Command |
|------|---------|
| Smoke (DCC) | `./scripts/run_smoke.sh` or `pytest projects/dcc -m smoke` |
| Regression | `./scripts/run_regression.sh` |
| Everything | `./scripts/run_all.sh` |
| Single file | `pytest path/to/test_file.py -v` |
| Single test | `pytest path/to/test_file.py::test_name -v` |
| HTML report | `pytest projects/dcc -m smoke --html=report.html --self-contained-html` |

## Adding another API

Read **[ADDING_NEW_PROJECT.md](ADDING_NEW_PROJECT.md)** — copy **`projects/dcc`** for a full pytest layout, or add **`projects/<slug>/contracts/`** plus `config/projects.yaml` if you only need the OpenAPI contract runner (Federation pattern).

## DCC OpenAPI contract suite (autogenerated)

The package **`framework.contract_runner.dcc.cli`** drives GET requests from **`projects/dcc/contracts/openapi.json`**: live discovery (`/subject`, `/sample`, …), case generation, then JSON + HTML reports.

- **Functional contract run:** `bash scripts/run_dcc_contract_suite.sh` or `python -m framework.contract_runner.dcc.cli`
- **Performance (positive cases only):** `bash scripts/run_dcc_perf.sh`
- **Browser UI:** `python launcher.py` (sets `DATACOMNS_DCC_BASE_URL` per selected environment)

Outputs: **`reports/dcc/contract/`** and **`reports/dcc/perf/`** by default — not the same files as pytest-html auto-reports.

## Memgraph vs API pairs (DCC)

1. Set `MEMGRAPH_DCC_URI`, `MEMGRAPH_DCC_USER`, `MEMGRAPH_DCC_PASSWORD` (see `.env.example`).
2. Author YAML under `projects/dcc/memgraph_validation/pairs/`.
3. Run: `bash scripts/run_dcc_memgraph_smoke.sh` or `pytest projects/dcc/tests/memgraph_api -m "memgraph_api and smoke" -v`
4. CI without DB: `DATACOMNS_SKIP_MEMGRAPH_DCC_TESTS=1`

## Print API / Memgraph results

Printing is **on by default**. Set **`DATACOMNS_PRINT_RESPONSES=0`** in `.env` to turn off `[DATACOMNS_PRINT]` logs. **`pytest.ini` includes `-s`** so that output appears in the terminal; use **`pytest --capture=sys`** when you want normal capture.

## Who to ask
